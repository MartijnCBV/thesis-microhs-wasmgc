\section{MicroHs}\label{sec:background:microhs}
MicroHs~\cite{microhs,microhsrepo} is a minimal Haskell compiler written in Haskell, with a runtime system written in C. MicroHs follows a standard compilation pipeline, where it starts by parsing the Haskell source code, then type-checking it, desugaring it into lambda terms, optimizing the lambda terms, and finally transforming the lambda terms into combinators through a process called bracket abstraction. MicroHs converts the combinators into C code, which is then combined with the runtime system into a single C codebase. This codebase can then be compiled with standard C compilers to obtain a native executable, or to Wasm using Emscripten~\cite{emscripten}. The compilation pipeline is illustrated in Figure~\ref{fig:microhs-pipeline}.

\paragraph{Combinators}
MicroHs uses a combination of combinators and primitive operations to represent compiled Haskell code. SKI combinators are a small set of combinators that can be used to represent any lambda term. The three basic SKI combinators are:
\begin{align*}
    S \; x \; y \; z & = x \; z \; (y \; z) \\
    K \; x \; y      & = x \\
    I \; x           & = x
\end{align*}
Further combinators include those used by Turner~\cite{turner} or are added by MicroHs itself to improve performance of the generated code. 

Primitive operations can be divided into two types: strict primitives and non-strict primitives. Strict primitives require all of their arguments to be reduced to weak head normal form (WHNF) before they can be applied, while non-strict primitives can be applied without reducing their arguments first. WHNF in the context of combinators means that the argument is either a primitive value (e.g. an integer) or a combinator or primitive application with to few arguments to be reduced. Examples of strict primitives include arithmetic operations such as addition and multiplication, while examples of non-strict primitives include the monad operations for the IO monad (\text{return} and \texttt{>>=}).

\paragraph{Representing Combinators}
To efficiently represent combinators and their arguments, MicroHs uses a graph representation. Each node can either be a leave node storing a combinator or a primitive value, or an internal node storing a reference to its left and right children. This allows for efficient sharing of subterms and avoids the need for copying when reducing combinators. For example, the combinator application \texttt{K (I x) y} can be represented as the graph shown in Figure~\ref{fig:combinator-graph}.

\paragraph{Runtime System}
After building the graph representation of the combinators, the runtime system is responsible for reducing the graph to WHNF. This is done by repeatadly walking the left spine of the graph until a leaf storing a combinator is reached, at which point the reduction rules for that combinator are applied. Since the reduction rules for most combinators involve multiple arguments, the runtime system maintains a left ancestor stack (LAS) to keep track of the ancestors of the current node (the left spine of the graph). This allows the runtime system to efficiently access the arguments needed for reduction without having to traverse the graph multiple times. An example reduction for the combinator application \texttt{K (I x) y} is shown in Figure~\ref{fig:combinator-reduction}.

We can use the algorithm shown in Algorithm~\ref{alg:graph_reduction} to perform the reduction. The \texttt{step} function takes the current node, the left ancestor stack, and the left ancestor stack pointer as input and returns the new current node and the updated left ancestor stack pointer after performing one reduction step. The algorithm continues until the current node is fully reduced (i.e. it is a leaf storing a primitive value or an application with to few arguments to be reduced).
\begin{algorithm}
  \caption{Graph reduction algorithm based on \cite{turner}.}
  \label{alg:graph_reduction}
  \SetKwInOut{Input}{Input}
  \SetKwInOut{Output}{Output}
  \SetKwComment{Comment}{/* }{ */}
  \Input{Node \texttt{n} to reduce}
  \Output{Reduced node}
  $las \gets \emptyset$\;
  $las_{pointer} \gets -1$\;
  $cur \gets n$\;
  \While{$cur$ is not fully reduced}{
      \eIf{$cur$ is a combinator}{
          $(cur, las_{pointer}) \gets \text{step}(cur, las, las_{pointer})$\;
      }{
          $cur \gets \text{leftChild}(cur)$\;
      }
      $las[las_{pointer}] \gets cur$\;
      $las_{pointer} \gets las_{pointer} + 1$\;
  }
\end{algorithm}

\section{WebAssembly}\label{sec:background:wasm}
Wasm is a binary stack-based instruction format. Wasm programs run in an isolated environment by construction. They can be interacted with from outside of the environment through imports and exports, this allows Wasm modules to call (imported) functions such as printing to the console, and allows a host environment to initiate the execution of a Wasm module by calling an exported function. The programs themselves are organized into modules containing functions, memory, imports/exports, and other components.

Wasm has a text format called WAT (WebAssembly Text format) which is a human-readable representation of the binary format. From now on we will use WAT syntax to illustrate Wasm concepts. Using WAT a simple list of instructions that adds two numbers looks like this:
\begin{minted}{wat}
(i32.const 1)
(i32.const 2)
(i32.add)
\end{minted}
The first two instructions push the constants 1 and 2 onto the stack, and the last instruction pops the two values from the stack, adds them together, and pushes the result (3) back onto the stack.

\paragraph{Instructions and Functions}
Instructions in Wasm are the basic building blocks of computation, including simple operations like \texttt{i32.add} for arithmetic and \texttt{local.get} and \texttt{local.set} for variable access. Control flow uses structured instructions such as \texttt{if}, \texttt{block}, \texttt{loop}, and \texttt{br} (branch), where conditions are represented as integers (0 is false, non-zero is true). Branching to a block label continues execution after the block end, while loop labels restart at the loop beginning.

Functions are blocks of code that can take parameters, return results, and contain local variables. They are defined using the \texttt{func} keyword and can be called with the \texttt{call} instruction.

An example piece of wasm code that demonstrates control flow and function calls is shown below:
\begin{minted}{wat}
;; import a print function
(import "env" "print" (func $print (param i32)))

(block $exit
  (loop $loop
    (local.get $counter)
    (i32.const 0)
    (i32.eq)
    (br_if $exit) ;; exit if counter == 0
    (local.get $counter)
    (i32.const 1)
    (i32.sub)
    (local.set $counter)
    (local.get $counter)
    (call $print)
    (br $loop) ;; continue looping
  )
)
\end{minted}

\paragraph{Types}
Wasm's type system consists of value types and heap types. Value types include primitive numeric types such as \texttt{i32} (32-bit integer) and \texttt{i64} (64-bit integer), and reference types. 

Reference types are references to heap types such as functions, structs, and arrays. The references are typed, meaning that every reference has a specific type associated with it. \texttt{anyref} is a supertype of all reference types in Wasm, allowing it to store references to any type of object, but it requires runtime type casts when accessing these fields. A cast can be performed using the \texttt{ref.cast} instruction, which casts a reference to a different type and traps if the cast is invalid. The \texttt{ref.test} instruction checks if a reference can be cast to a given type. Reference types can also be nullable, meaning that they can point to either an object of the specified type or null. The \texttt{ref.null} instruction creates a null reference of a given type, and the \texttt{ref.is\_null} instruction checks if a reference is null.

Custom types can be defined in Wasm using the \texttt{type} keyword. These custom types can be used to define more complex data structures such as structs and arrays. Structs are collections of named fields, each with their own type, while arrays are collections of elements of the same type. For example, we can define a struct type for a binary tree node as follows:
\begin{minted}{wat}
(type $Node (struct
  (field $left  (mut anyref))
  (field $right (mut anyref))
))
\end{minted}
In this definition, we use the \texttt{anyref} type for the fields, which allows us to store references to any type of object in the \texttt{left} and \texttt{right} fields of the \texttt{Node} struct. The \texttt{mut} keyword indicates that the fields are mutable, meaning that their values can be changed after the struct is created.
