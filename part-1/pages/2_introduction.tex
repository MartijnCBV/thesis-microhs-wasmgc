\chapter{Introduction}\label{ch:introduction}
Today, porting Haskell code to web browsers is complicated. Current approaches such as transpilation to JavaScript~\cite{ghcjs} suffer from performance issues and lack of semantic similarity (e.g. functional vs imperative, lazy vs eager eveluation, etc.), while solutions such as containerization can not be used in browser environments.

A promising alternative is compiling Haskell to WebAssembly (Wasm)~\cite{wasm,ghcwasm,asterius}, a low-level bytecode format designed for efficient execution in portable environments such as web browsers, but also for non-browser engines which can be used for server-side applications. Wasm however has its own challenges, the original version of Wasm (WasmMVP) does not natively support garbage collection (GC), which forces compilers targeting WasmMVP to ship their own memory management systems. 

This changed with the recent release of Wasm 3.0~\cite{wasm3} due to the integration of the WasmGC proposal, which extends Wasm with user-defined garbage collected (GC) data types. This allows compilers targeting Wasm to make use of the host environment's GC instead of shipping their own, which can lead to smaller binary sizes and improved maintainability. This is due to the fact that runtime systems can be simpler and smaller as they do not need to include a GC implementation.

Despite the potential benefits of using WasmGC, current Wasm backends of compilers such as GHC's Wasm backend~\cite{ghcwasm} do not make use of it, and instead target WasmMVP. 

\paragraph{Contributions} The goal of this project is to investigate the performance tradeoffs of using WasmGC as a target for Haskell compilers. To this end we will extend the Haskell compiler MicroHs~\cite{microhs} with a WasmGC backend. Concretely, we will do the following:
\begin{enumerate}
    \item Design and implement a custom domain specific language (DSL) for generating WasmGC code, for use in the runtime system and backend of the compiler. This will help us to write more maintainable and readable code compared to writing raw WasmGC.
    \item Design and implement a runtime system for the WasmGC backend using the custom DSL.
    \item Evaluate the produced Wasm binaries in terms of size and runtime performance against other Haskell-to-Wasm compilers and MicroHs's existing C backend.
\end{enumerate}

\paragraph{Previous Work}

Previously, Anand~\cite{anandthesis} made a prototype extension to MicroHs to target WasmGC and implemented a proof-of-concept runtime system written in a combination of WasmGC and a custom DSL. Initial experiments showed promising results in terms of binary size, however runtime performance was not tested. Our work will extend this work with a new DSL capable of generating the full runtime system instead of just parts of it, building a more complete runtime system that supports more Haskell features, and evaluating the runtime performance of the generated Wasm binaries.
